# Contributing to Serenity

Thank you for your interest in contributing. Before submitting any contribution, please
read this document carefully. By submitting a contribution you agree to the terms below.

---

## 1. Contributor Licence Agreement (CLA)

documentation, bug reports, feature requests, translations, or skill files — you agree

  as part of Serenity without further permission from you
- You retain no proprietary rights in your contribution once submitted

### 1.2 Your Representations

By submitting a contribution, you represent and warrant that:

- You are the sole author of the contribution, or you have the legal right to submit it
- The contribution does not infringe any third-party patent, copyright, trademark, or
  other intellectual property right
- The contribution does not contain any confidential information belonging to a third
  party or your employer
- If your contribution includes code from third parties, you have disclosed this and
  the relevant licence terms are compatible with this project

### 1.3 Why This Matters

Serenity is dual-licensed (free for personal use, paid for commercial use). For this
model to be legally sound, the copyright in all contributions must be unified. Without
clear copyright assignment, contributors could later claim rights that conflict with the
project's commercial licence structure.

## 2. What We Accept

We welcome contributions including:

- Bug fixes with clear reproduction steps
- Performance improvements backed by benchmarks
- New tools or skills that are broadly useful
- Documentation improvements
- Test coverage

We may decline contributions that:

- Add unnecessary complexity or dependencies
- Change core architecture without prior discussion
- Include AI-generated code that has not been reviewed and tested by the contributor

---

## 3. How to Contribute

1. **Open an issue first** for significant changes — discuss before building
2. **Fork** the repository and work on a feature branch
3. **Write tests** for any new functionality where applicable
4. **Submit a pull request** with a clear description of what you changed and why
5. **Sign off** your commit with `git commit -s` to indicate acceptance of this CLA

---

## 4. Code Standards

- Follow the existing code style (Python: PEP 8, typed where the codebase uses types)
- Keep pull requests focused — one concern per PR
- Do not include `.env` files, API keys, or personal credentials
- All new tools must have a `description` property that clearly explains their purpose

---

## 5. No Obligation

Submitting a contribution does not obligate Daniel T Niamke to include it in Serenity.
All contributions are reviewed at our discretion.

---

## 6. Conduct

All contributors are expected to engage respectfully. Harassment, abuse, or hostile
behaviour of any kind will result in permanent exclusion from the project.
